# `/api/inngest` 500 Fix Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix the `/api/inngest` route returning HTTP 500 in production by polyfilling `globalThis.DOMMatrix` (required by `pdfjs-dist` at module load) via `@thednp/dommatrix` and lazy-importing `pdf-parse` in the parser.

**Architecture:** Three changes: (1) add `@thednp/dommatrix` as a direct dependency, (2) create a side-effect-only shim module that polyfills `globalThis.DOMMatrix` and patches the missing `*Self` methods onto its prototype, (3) refactor `src/lib/parsers/pdf.ts` to dynamically import the shim and `pdf-parse` inside the function body, decoupling `/api/inngest` cold start from `pdfjs-dist` evaluation.

**Tech Stack:** TypeScript, Next.js 16 App Router, Vitest, Vercel Lambda (Node 20.x), `pdf-parse@2.4.5`, `pdfjs-dist@5.4.296` (transitive), `@thednp/dommatrix@^3.0.4`.

**Spec:** `docs/superpowers/specs/2026-06-09-pdfjs-dommatrix-shim-design.md`

- [ ] **Step 3: Create the shim file with exact contents**

Create `src/lib/parsers/dommatrix-shim.ts` with the following exact contents:

```ts
// @thednp/dommatrix ships as CJS: `module.exports = Class`. TS's
// `esModuleInterop` unwraps this to `import X from 'pkg'` for us,
// once the ambient declaration in src/types/dommatrix.d.ts provides
// the default export.
import DOMMatrix from '@thednp/dommatrix'

declare global {
  var DOMMatrix: typeof DOMMatrix
}

// Augment the prototype with the methods pdfjs-dist uses that
// @thednp/dommatrix doesn't implement (no-op chains: identity
// transform). @thednp/dommatrix already implements multiplySelf,
// translateSelf, scaleSelf, rotateSelf and friends. The actual
// gaps for pdfjs-dist's text-extraction path are:
//   - invertSelf()        (pdf.mjs:7622, 16577, 16607, 16686, 16792)
//   - preMultiplySelf()   (pdf.mjs:16697)
//   - invert()            (defensive; pdfjs-dist doesn't currently
//                          call this on a fresh matrix, but it appears
//                          in related APIs)
if (typeof DOMMatrix.prototype.invertSelf !== 'function') {
  DOMMatrix.prototype.invertSelf = function () { return this }
}
if (typeof DOMMatrix.prototype.preMultiplySelf !== 'function') {
  DOMMatrix.prototype.preMultiplySelf = function () { return this }
}
if (typeof DOMMatrix.prototype.invert !== 'function') {
  DOMMatrix.prototype.invert = function () { return this }
}

if (typeof globalThis.DOMMatrix === 'undefined') {
  globalThis.DOMMatrix = DOMMatrix
}

export {}
```

- [ ] **Step 2: Replace the static import with dynamic imports**

Replace the entire file `src/lib/parsers/pdf.ts` with the following exact contents:

```ts
export async function parsePdf(buffer: Buffer): Promise<string> {
  // Side-effect import: polyfills globalThis.DOMMatrix before pdf-parse
  // (and its transitive pdfjs-dist) evaluates on first use. This must
  // run before the dynamic import of pdf-parse below.
  await import('./dommatrix-shim')
  const { PDFParse } = await import('pdf-parse')

  const parser = new PDFParse({ data: new Uint8Array(buffer) })
  try {
    const result = await parser.getText()
    return result.text ?? ''
  } catch (error) {
    console.error('Error parsing PDF:', error)
    throw new Error('Failed to parse PDF file.')
  } finally {
    await parser.destroy().catch(() => {})
  }
}
```

Key change: line 1 (`import { PDFParse } from 'pdf-parse'`) is removed. The `import` happens inside the function body via two sequential `await import(...)` calls — first the shim, then `pdf-parse`. This decouples module load of the inngest route (and any other module that imports `parsePdf` statically) from `pdf-parse` evaluation.

## Troubleshooting

### Test fails after dynamic import change

If `npx vitest run src/lib/parsers/pdf.test.ts` fails with errors like "Cannot find module 'pdf-parse'" or "PDFParse is not a constructor":

1. Check that `vi.mock('pdf-parse', ...)` is at the top of the test file (above the `import { parsePdf }` line). It is — verify by reading the file.
2. Check that the mock factory returns `{ PDFParse: MockPDFParse }` (not the default export). It does — verified.
3. If still failing, the issue may be that vitest's dynamic-import hoisting doesn't intercept `await import('pdf-parse')` inside a function. The fix is to change the mock to use `vi.doMock` or to mock `./pdf` instead of `pdf-parse`. But this should not be needed; `vi.mock` is supposed to intercept all imports including dynamic.

### Vercel build fails with "Cannot find module '@thednp/dommatrix'"

Add to `next.config.ts` under `experimental`:
```ts
transpilePackages: ['@thednp/dommatrix'],
```

Then re-deploy.

### `/api/inngest` still 500s after deploy

Check the Vercel function logs (not just the HTTP response). Look for:
- `ReferenceError: DOMMatrix is not defined` — the shim isn't being applied. Verify `parsePdf` is actually being called (it shouldn't be on the GET probe). The 500 would mean the import chain is still evaluating pdf-parse at module load. Check that `processDocument` doesn't have a side-effect import of `parsePdf` at module top.
- `TypeError: ...X... is not a function` — the shim is missing a method that pdfjs-dist uses. Add it to the prototype patches.
- `MODULE_NOT_FOUND` — the install didn't include the new dep. Re-run `npm install` and re-push.

### `@napi-rs/canvas` warning still appears in logs

Expected and harmless. The warning means pdfjs-dist tried to load canvas, failed, and fell back to using `globalThis.DOMMatrix` (which our shim provides). The text extraction path works.
